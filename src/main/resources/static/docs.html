How Lumina Crawl works

Lumina Crawl is a Spring Boot application that crawls websites with Jsoup, runs a pipeline of quality analyzers, persists reports, and optionally calls Google Gemini for actionable recommendations.

Architecture Crawler Analyzers Scoring Gemini AI API Limits

System architecture

The app serves a static SPA from src/main/resources/static and exposes REST endpoints under /api. Analysis is orchestrated by AnalysisOrchestratorService, which coordinates crawling, per-page analysis, aggregation, and persistence.

Tech stack

  • Java 17 + Spring Boot 3.2 (Web, JPA, Validation)
  • Jsoup — HTML fetch and DOM parsing (no JavaScript execution)
  • RestTemplate — sitemap fetch, link HEAD probes, Gemini HTTP calls
  • Gemini API — structured JSON recommendations with local fallback
  • H2 (dev) or MySQL (production) for report history

Web crawler

WebCrawlerService performs bounded graph traversal starting from the seed URL. Links are normalized via UrlNormalizer (scheme, host, path, fragment stripping, invalid percent-encoding sanitization).

Traversal strategies

  • BFS — breadth-first queue; discovers pages level by level (default).
  • DFS — depth-first stack; goes deep along one branch before backtracking.

Sitemap & bot protection fallback

JavaScript-heavy and Cloudflare-protected sites often return few HTML links. When the crawl queue is sparse, SitemapDiscoveryService reads robots.txt and XML sitemaps (via RestTemplate + regex <loc> extraction) to seed additional URLs. BotProtectionDetector flags 403 responses and JS-challenge pages; warnings are surfaced in the report.

SPA limitation: Jsoup only parses server-rendered HTML. React/Vue/Angular routes that require a browser runtime are not executed — sitemap seeding is the primary workaround for discovering URLs on those sites.

Analyzer pipeline

After crawling, each page passes through specialized analyzers. Results are merged into a single QualityReportDto.

Module What it checks
UrlAnalyzerService URL length, slug readability, query params, special characters
UrlFilterService Regex rules — login, cart, checkout, session IDs, admin, PDFs
HtmlSeoAnalyzerService Title, meta description, H1/H2, image alt attributes
ContentQualityService Word count, keyword density, Flesch readability, thin content (<300 words), Jaccard duplicate similarity
BrokenLinkCheckerService HEAD requests on internal/external links (capped, 4s timeout)
SiteTreeBuilderService Crawl hierarchy tree from parent map or path-based fallback

Regex URL filter examples

  • .*login.* — login pages
  • .*cart.* / .*checkout.* — e-commerce flows
  • .*\?sessionid=.* — session parameters
  • .*wp-admin.* — CMS admin panels
  • .*\.pdf$ — static PDF assets

Quality scoring

Each page receives sub-scores for SEO, URL structure, and content quality. The orchestrator computes a weighted overall score (0–100) from crawl-wide aggregates: broken link ratio, thin content share, duplicate pairs, and average per-page scores.

Score rings in the UI use thresholds:

  • 70+ — green (healthy)
  • 40–69 — amber (needs attention)
  • <40 — rose (critical issues)

Gemini AI engine

GeminiAiService sends a condensed crawl summary to the Gemini generateContent API (default model: gemini-3.1-flash-lite). The prompt requests a JSON array of recommendations with priority, category, and description.

If the API key is missing, the request times out, or parsing fails, a local fallback generates rule-based suggestions from detected issues (missing meta, broken links, thin content, etc.).

Configure via .env or environment variables:

GEMINI_API_KEY=your_key_here GEMINI_MODEL=gemini-3.1-flash-lite

REST API

Method Endpoint Description
POST /api/analyze Run full site analysis
GET /api/reports List recent saved reports
GET /api/reports/{id} Load a report by ID
GET /api/health Health check for deploy probes

Example request

curl -X POST http://localhost:8080/api/analyze \ -H "Content-Type: application/json" \ -d '{ "url": "https://example.com", "maxPages": 30, "maxDepth": 4, "crawlStrategy": "BFS" }'

Request fields

  • url — must start with http:// or https://
  • maxPages — 5–100 (default 15)
  • maxDepth — 1–8 (default 4)
  • crawlStrategy — BFS or DFS

Performance & safety limits

  • External broken-link checks capped at 25 unique targets
  • Uncrawled internal link checks capped at 30
  • Link probes use HEAD with a 4s timeout
  • RestTemplate connect/read timeouts: 10s / 30s for Gemini
  • Frontend fetch timeout: 3 minutes for long crawls

These limits prevent hangs on large sites (e.g. openai.com) while still producing actionable reports within a reasonable window.
